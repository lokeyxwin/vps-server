# 0002. 纳管模式由 XrayWorker 包揽（不另起 TakeoverWorker）

**日期**:2026-06-06
**状态**:Accepted

## 背景

传进来的服务器不一定是"空白"。在 ADR-0001 工人架构下,XrayWorker 上去后会
遇到 4 种现状:

| 现状代号 | xray 装了 | xray 在跑 | 有出口配置 | 业务含义 |
|---------|----------|----------|----------|---------|
| A 全新 | ❌ | – | ❌ | 全新空白服务器,要装 |
| B 已装停了 | ✅ | ❌ | ❌ | 装过但服务挂了,要启 |
| C 已装跑着空配置 | ✅ | ✅ | ❌ | xray 跑着但没挂出口,啥都不做 |
| D **纳管** | ✅ | ✅ | ✅ | 别人挂过出口,要接管 |

情况 D 是关键 —— 如果不把别人挂的出口信息反推到我们的表里直接标 running,
ProxyDeployWorker 来挑端口时会撞到"未知占用",或者把客户端正在用的端口踢掉。

## 决策

### 1. XrayWorker 内部分 4 个分支处理 A / B / C / D

不另起 TakeoverWorker。锁不换手、SSH 不重连、职责连贯。

### 2. 纳管时不迁移原有端口

客户端可能正在使用别人挂的端口(比如 1080),**强行迁移到 18441-18450 会断生产**。
纳管的策略是:

- 把别人挂着的 inbound / outbound / 路由规则**原样接管**(直接读 + 直接入表)
- 端口保留原位置(1080 还是 1080 / 8888 还是 8888 / ……)
- 不替用户做端口规划改动,只做"事实建档"

### 3. 端口策略改为"排除清单"

放弃原 "限定 18441-18450 业务段" 规则(由 ADR-0001 沿用旧 CLAUDE.md 继承)。
新规则:

- 维护一份**必须避开的常用端口清单**(0-1023 well-known + 几十个高频应用,
  例如 1080 SOCKS / 3306 MySQL / 6379 Redis / 8080 / ...)
- 在排除清单之外的高位端口都可用(由 ProxyDeployWorker 随机挑)
- 端口排除清单实现位置:`config.py::EXCLUDED_PORTS`(或单独 `kits/install_xray/ports.py`)
- 配置初稿留待用户审,落地见后续配套任务

> **注意**:纳管端口(用户已经在用的)即使不在我们排除清单里,**也直接接管不动**;
> 排除清单只约束"新分配端口"和"我们自己的默认入口"。

### 4. VPS 表字段语义改变

| 旧字段 | 含义 | 状态 |
|--------|------|------|
| `idle_port_count` | 空闲端口数 | **删除** |

| 新字段 | 含义 | 由谁写 |
|--------|------|--------|
| `used_port_count` | 已配置且**内 ping 通**的代理出口数 | XrayWorker 纳管完成时写;后续 ProxyDeployWorker 配新出口时 +1 |

### 5. 纳管 IP 入库时 `expire_date` 字段为 null

纳管反推出来的上游 IP 凭据,**不知道什么时候到期**——这是项目从外部世界拿到的
未知数据。

- 纳管 IP 入 `ip_record` 时 `expire_date = null`
- IP 表的 `expire_date` 字段必须改 nullable(若当前不是)
- 巡检(ExpiryWorker)看到 null 直接**跳过**,不当过期处理
- 用户后续手动确认到期日时再 update

### 6. 内 ping 不通的纳管 IP

纳管 6 步流程里,XrayWorker 对每条反推出来的 inbound 做内 ping:
- **通** → 算"正常使用中" → 计入 `used_port_count` + IP 入库 status=active
- **不通** → 大概率上游 IP 已过期 → IP 入库标记 `is_active=0`
  + 留 expire_date 空白 + 备注"纳管时内 ping 不通,疑似过期"
  + (具体过期判定/标日期等到后续巡检模块再深化)

## 备选方案

### 方案 Y:单独 TakeoverWorker (被否决)

XrayWorker 干 A / B / C,情况 D 派 `vps_task=takeover` 给 TakeoverWorker。

**否决理由**:
- 换锁复杂:XrayWorker 干完后释放锁,TakeoverWorker 来抢同一台,锁过程容易乱
- SSH 重连费时:已经建好的 SSH 连接被关掉,再开新的浪费几秒
- 职责连贯性差:"让 xray 服务在跑 + 让 DB 知道这台机全貌"本质是一件事,
  拆给两个工人反而破坏了原子性

### 方案 Z:把所有纳管端口迁移到 18441-18450 段 (被否决)

读出旧 inbound → 删旧 → 在 18441-18450 段重建。

**否决理由**:
- **断生产**:客户端可能正在用旧端口,删旧 inbound 会直接掐流量
- 客户端不归我们管,无法通知"端口换了"
- 强迁的 ROI 几乎为零(端口分布美观不值得断生产)

## 后果

### 好处

- XrayWorker 一锅端,锁/SSH 复用,流程清晰
- 纳管不影响用户正在跑的业务(零打扰)
- 端口策略灵活(排除清单可演化,不绑死硬编码段)
- IP 表能正确表达"我们知道的过期日"vs"未知"两种状态

### 引入的新约束

- 端口排除清单必须维护(常用端口的初稿见后续 task)
- VPS 表的 `idle_port_count` 字段必须**删除**或重命名,引入 `used_port_count`
- IP 表 `expire_date` 必须 nullable(已经是,无需改)
- 巡检/到期判断逻辑里必须**显式处理 null**(跳过)
- ProxyDeployWorker 挑端口的逻辑变了:
  - 旧:在 18441-18450 段找空位
  - 新:在 1024-65535 里随机挑 + 排除"清单端口" + 排除"该 VPS 已用端口"
- 旧 CLAUDE.local.md 里"事实表模式:每台 VPS 最多 10 条对应 18441-18450"
  规则**被本 ADR 取消**

### 风险

- 排除清单不全 → 随机分到 MySQL 3306 之类的端口,被系统服务占着
  缓解:首版清单宽松一点(把常见 well-known + 高频应用都列上),后续补
- 纳管反推可能不全(老版本 xray config 格式变化)
  缓解:已有旧代码 `xray.config.extract_port_bindings` 沿用,有问题再说
- 内 ping 不通就判过期不一定准(可能只是网络抖动)
  缓解:首版只标记疑似过期 + 等手动确认;不主动删数据

## 用户口述原话(关键节选)

> "刚刚我没有考虑到的点是,如果把原有配置搬到我们限定的端口内的话,那用户用户
> 已经在使用的配置,他就用不了了呀。这样子会影响生产环境啊。要不把所,就是限定
> 端口给去掉吧,就直接纳管就好了,这样子就直接 add 进来。"

> "然后空闲端口数的话,就不需要统计了。就是上去之后,我们先我们先限定一些,就是
> 常用的端口,就是已经被特殊定义的端口,就一个完整的合集,看看去哪里找吧。"

> "VPS 上面的可用端口数改成已用端口数,已用的定义就是配置了多少条代理,还正在
> 正常使用的。"

> "纳管进来,然后去统计这些数量。我看看这个数据给谁写先。给初始化的那工人写,他
> 就去找这台服务器,然后去计算这台配置里面有多少个端口它是能够正常使用的。"

> "只要从内聘能聘的通,就代表这个这个代理出口它是能够正常去使用的"

> "写进表里面的时候,就直接写未知,对,未知,那就知道这个就是已经配置好了纳管的
> IP,但是他没有没有没有到期日期,不知道,所以就对,然后需要等用户去那边说续费了
> 什么再说"

> "巡检的时候就直接跳过就行了。然后如果内聘不通,内聘不通的话,大概率就是,如果
> 内聘不通且这条配置已经在 IP 表里面,那就代表着啊这条 IP 已经过期了"
