# 0003. XrayWorker 简化为 3 分支 + 统一收尾(修订 ADR-0002 的 4 分支决策)

**日期**: 2026-06-06
**状态**: Accepted
**修订**: 取代 ADR-0002 §决策 §1(4 分支 A/B/C/D)中的"D 纳管为特殊路径"部分。
        ADR-0002 其余决策(端口策略、used_port_count、纳管 IP expire_date=null 等)**全部保留有效**。

## 背景

ADR-0002 把 XrayWorker 处理服务器时按现状分 4 个分支:
- A 全新空白 → 装+起+自启
- B 已装停了 → 起
- C 已装空配 → 啥不做
- D **纳管(有别人挂的出口)** → 抠配置 + 入 ip/proxy_record + 补默认入口

实施中跟用户对齐时,用户提出更清晰的简化方式:

> "如果没有安装的,那他就从头干到尾,安装、常驻、拉起、配置、检查一系列跑通。
>  然后如果安装了,那上去那就是先检查有没有运行,有运行。OK"

> "ABC 全都查配置纳管"

意思: **纳管不应该是"D 分支的特殊场景",而是所有路径的必经收尾步骤**。
因为无论 ABC 哪种现状,只要存在出口配置就该接管;没有就跳过,
不需要为此独立一个"D 分支"。

## 决策

### 1. 取消"分支 D 纳管"独立路径

把旧 D 分支的 6 步合并为"统一收尾 7 步",ABC 三分支跑完都走这段:

```
共同前置: 看 xray_version + is_running 决定走 A/B/C
   ↓
分支 A 全新空白(xray_version 空):
  ① 装 ② 起 ③ 自启 ④ 验证
分支 B 已装停了(xray_version 有 + 没跑):
  ① 起 ② 自启 ③ 验证
分支 C 已装跑着(xray_version 有 + 跑着):
  (啥都不干)
   ↓
统一收尾 7 步(所有分支必经):
  ① 读配置 read_config
  ② 抠出现有出口配置 extract_existing_outbounds  ⭐ 抠信息类
  ③ 如果 ② 返回空 → 跳过 ④⑤;否则逐条内 ping
  ④ 写 ip_record(每条上游一行,纳管 IP expire_date=null)
  ⑤ 写 proxy_record(每个出口端口一行)
  ⑥ 补默认入口 18440(没有则加)
  ⑦ upload_config → validate → reload → is_running 验证
   ↓
出口: vps.stage='running' + xray_version + used_port_count + task.status='done'
```

### 2. 决策点 vs 收尾的清晰分界

- **分支决策**只看现状(xray_version + is_running),决定**前置装/起步骤**
- **收尾步骤**对每个 VPS 都执行,不依赖现状
- 配置里有出口 → 接管(②~⑤ 写库);没出口 → 跳过写库,但仍执行⑥⑦保证默认入口存在

## 备选方案

### 方案 X(被否决): 保留 4 分支独立路径(ADR-0002 原方案)

**否决理由**:
- 实际实现时 ABC 都必须读配置确认现状,跟 D 的 "抠出口" 步骤重复
- 4 分支版的"判断有没有别人挂的出口"在 ABC 时也必须做(否则错过纳管)
- 简化为统一收尾后,代码量减少,逻辑路径更清晰

### 方案 Y(被否决): 取消 ABC 三分支区分,全部走完整流程

**否决理由**:
- 没必要给已经跑着的 VPS 重复跑装/起,浪费资源
- 前置步骤的差异(装 vs 起 vs 不动)有实际价值,值得保留

## 后果

### 好处

- 实现代码减少:取消 D 独立路径,合并到统一收尾,内部代码量降低
- 逻辑清晰:分支决策(前置) vs 统一收尾(后置)职责分明
- 不再担心"漏纳管":统一收尾保证每个 VPS 完工时都接管现有配置
- 测试更简单:测分支决策(看现状走对方向) + 测统一收尾(读+抠+ping+写)
  两块独立,组合即可

### 引入的新约束

- XrayManager 需要确保 `extract_existing_outbounds()` 在**空配置**情况下也
  返回 `[]`(不抛错),给统一收尾的 ③ 步判断空配置走

- 测试用例需要覆盖:
  - 分支 A 后接管空配置(常见场景:新装机)
  - 分支 A 后接管非空配置(罕见,理论上不可能,但代码要兜底)
  - 分支 B 后接管空配置
  - 分支 B 后接管非空配置(真实业务场景:之前装过+挂过+服务挂了)
  - 分支 C 后接管空配置(已装跑着但没出口)
  - 分支 C 后接管非空配置 ⭐ (这是 ADR-0002 旧"D 分支"的核心场景)

### 风险

- 简化后没了"D 分支"标识,文档里描述"纳管"时容易混淆
  缓解:spec.md §3 明确写"统一收尾 = 纳管 + 默认入口 + reload"
- 实现者可能错把 ABC 任一分支视为"不要做收尾"
  缓解:任务单实现轮廓里强调"无论哪个分支,都必走统一收尾 7 步"
