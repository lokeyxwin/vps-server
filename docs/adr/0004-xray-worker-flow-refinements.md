# 0004. XrayWorker 流程补细化(supersede ADR-0002 §3/§6,补 ADR-0003)

**日期**: 2026-06-07
**状态**: Accepted
**Supersedes(局部)**:
- ADR-0002 §决策 §3 "默认入口端口固定 XRAY_DEFAULT_PORT(=18440)" 那部分 → 改让步算法
- ADR-0002 §决策 §6 "内 ping 不通的纳管 IP" 全段 → 改为直接 remove 三件套,不记表

**补充(非推翻)**:
- ADR-0003 §决策 §1 流程图中分支 B/C → 各补"看自启没设就设"前置步

ADR-0002 其余决策(§1 XrayWorker 包揽纳管 / §2 纳管端口不迁移 / §3 排除清单 / §4 used_port_count / §5 纳管 IP expire_date=null)**全部保留有效**。

ADR-0003 §决策 §1 的 3 分支 + 统一收尾原则**保留有效**,本 ADR 只补流程细节。

---

## 背景

ADR-0001 / 0002 / 0003 落定后,跟用户深入对齐 XrayWorker 上场后具体怎么干活时,发现需要补 / 改 4 个点:

1. **ADR-0003 的"装了跑着""装了没跑"分支漏了"自启检查"** —— xray 在跑不代表设过 enable,VPS 重启就丢
2. **18440 默认入口可能被别人占着** —— ADR-0002 §3 写"固定 18440",没考虑别人挂了 18440 → 上游代理这种场景
3. **内 ping 不通的纳管出口处理过保守** —— ADR-0002 §6 标 `is_active=0` 留尾巴,实际上"连不通的配置就是废了",留尾巴价值低
4. **"直进直出"判定标准没明定** —— spec.md 隐式假设默认入口 = 端口 18440,实际上更准确的本质是"socks5 → freedom 出本机 IP"组合,跟端口无关

---

## 决策

### 1. ADR-0003 §1 分支 B/C 各补"看自启没设就设"前置步

补充后的流程:

```
分支 A 全新空白(xray_version 空):
  ① 装 ② 起 ③ 自启 ④ 验证

分支 B 已装停了(xray_version 有 + 没跑):
  ① 看自启没设就设  ⭐ 新增
  ② 起
  ③ 验证 is_running

分支 C 已装跑着(xray_version 有 + 跑着):
  ① 看自启没设就设  ⭐ 新增
  (其他啥都不干)
```

**理由**: xray 服务在跑只代表当前进程活,不代表 systemd `enable` 设过。VPS 重启后 xray 不会自启,业务断流。补"看自启没设就设"是闭环必需。

**新增工具依赖**: `XrayManager.is_enabled()` 查询方法(本 ADR 引出,落在 task T-03 占位 + T-07 真实现)。

---

### 2. 默认入口端口让步算法(supersede ADR-0002 §3 "固定 18440")

**旧规则**:默认入口端口固定 `XRAY_DEFAULT_PORT = 18440`。

**新规则**:

```
首选 18440
   ↓ 被占(被别人挂的"非 freedom" inbound 占着)
   ↓
试 18439
   ↓ 还被占
   ↓
试 18438
   ↓ ...
   ↓
下限 1024(不进 well-known 段)
   ↓ 一路降到 1024 都被占
   ↓
任务失败 (task.status='failed', last_error_code='no_default_port')
等人工介入
```

**理由**:
- 别人挂了 18440 → 上游代理,这条不是"直进直出",不能借用它当默认入口
- 强行 remove 别人的 18440 → 上游代理就是断生产,跟 ADR-0002 §2 "不断生产"原则直接冲突
- 让步几次找新端口最不打扰别人

**实现要点**: 每次降一位前都要看 "这个候选端口是不是已经被任何 inbound 占着(不管是 freedom 还是其他 outbound)"。

---

### 3. "直进直出"判定标准(原 ADR / spec 没明定)

**判定**:

> inbound 协议 == `socks5` **且** 路由到的 outbound 协议 == `freedom`。
> 端口不限。

**效果**:
- 配置里任何符合此判定的 inbound → 它**就是**"直进直出"
  - 不入 `ip_record`,不入 `proxy_record`(不是代理资产,只是 xray 启动用的内部组件)
  - 不走统一收尾的"内 ping → 写库 / remove"流程
- 配置里**存在**任何"直进直出" inbound → **不补**新的默认入口(借用现有的就行)
- 配置里**不存在**任何"直进直出" inbound → 补一条,端口按 §决策 §2 让步规则确定

**理由**:
- "直进直出"本质 = "不走任何代理,直接从本机 IP 出去",跟端口号无关
- 用户业务上 99% 是"经代理跳板出去",别人挂"自己用的本地直连跳板"虽然罕见但要兜底
- 判定靠 outbound 类型 = freedom(xray 内置直出),最准确

---

### 4. 内 ping 不通的纳管出口 → 直接 remove 三件套(supersede ADR-0002 §6)

**旧规则**(ADR-0002 §6):

```
内 ping 不通 → IP 入库标记 is_active=0
            + 留 expire_date 空白
            + 备注"纳管时内 ping 不通,疑似过期"
            + 巡检模块后续处理
```

**新规则**:

```
内 ping 不通 → 直接调 remove_proxy_binding(vps_port) 删三件套
            → ip_record / proxy_record 完全不记这条
            → 不留任何尾巴
```

**理由**:
- 内 ping 不通 = 服务器自己连自己都不通 = 客户端连了也必然不通 = 这条配置实际已废
- remove 它**不会断任何正在跑的生产**(它本来就跑不动)
- 留 `is_active=0` 的尾巴价值低(用户不会回头处理 expire_date=NULL 的一堆"疑似过期"条目)
- 留着废 inbound 占端口 + 让 xray 配置更乱
- `remove_proxy_binding` 现有实现已经够智能(共享 outbound 场景自动保留,见 §决策 §5)

---

### 5. 共享 outbound 场景的 remove 自然兜底(新约定,不 supersede 旧的)

xray 配置里 inbound 和 outbound 是 N:M 关系(通过路由规则关联)。**别人挂的代理**可能出现"多个 inbound 共享同一个 outbound" 的形态(同一条上游代理给多个客户端共用)。

`xray/config.py::remove_proxy_binding` 现有实现的 "防御性删 outbound" 逻辑天然兜住这种场景:

```
逐条处理纳管出口, 每条内 ping → 不通就调 remove_proxy_binding(vps_port):

   情况 A: 共享 outbound 的其他 inbound 还通
          → 工具删本条 inbound + 本条路由
          → 工具看 outbound 还被其他路由引用 → 保留 outbound
          → 通的那条 inbound 继续服务客户端, 不断流 ✅

   情况 B: 共享 outbound 的所有 inbound 都不通(上游死了)
          → 逐条 remove, 每条都看"outbound 还被剩下的引用吗"
          → 最后一条 remove 时, outbound 引用归零 → 工具自动连带删
          → 整组三件套清干净 ✅
```

**已知风险(已被用户看见并接受)**:

多个 inbound 共用同一个 outbound 时,如果它们**同时**配错账密(罕见,但不是不可能),整组会被误判为"上游死了"全删。当前业务规模下这种场景极罕见,接受此风险。

---

### 6. 默认入口监听 0.0.0.0 + noauth(原 ADR / spec 未明定)

**新规则**:

- 监听地址 `0.0.0.0`(全网监听,不绑 127.0.0.1)
- 协议 `socks5`
- 无账密(`noauth`)

**配合防火墙做扫描防护**:

- 防火墙端口豁免列表**只放业务代理端口**(由后续 ProxyDeployWorker 分配的高位端口)
- 默认入口端口(18440 或让步后的端口)**不在防火墙豁免列表**里
- 外部扫描到 `<vps_ip>:18440` 也连不上(被防火墙拦在外)
- 服务器内部 ping(走 localhost / 127.0.0.1)不受防火墙影响,正常工作

**理由**:
- 用户拍 0.0.0.0,接受"监听地址全开"
- 安全靠"防火墙不放行端口"兜,等价于实际外部不可达
- 比 127.0.0.1 简单(不用在监听地址和"客户端会不会从外部连"之间切换)

## 备选方案

### 方案 a:默认入口固定 18440,占用直接抢(被否决)

强行 `remove_proxy_binding` 别人挂在 18440 的代理,再 `add_proxy_binding` 18440 → freedom。

**否决理由**:
- 跟 ADR-0002 §2 "不断生产" 直接冲突
- 别人的客户端正在用 18440 → 强 remove → 立刻断流
- 客户端不归我们管,无法通知"端口换了"

### 方案 b:内 ping 不通仍按 ADR-0002 §6 标 `is_active=0`(被否决)

留尾巴 + 备注"疑似过期",等手动确认。

**否决理由**:
- 用户判断"留尾巴等人看"实际价值低 —— 用户不会回头处理一堆 `expire_date=NULL` 的疑似过期条目
- 留废 inbound 占端口 + 让 xray 配置更乱
- `remove_proxy_binding` 智能版已经够安全(共享场景自动保留),不会误伤生产
- "不记 = 干净,记 = 噪音" 的取舍下,选不记

### 方案 c:监听 127.0.0.1(被否决)

默认入口绑 127.0.0.1,只接受本机内部连接。

**否决理由**:
- 用户拍 0.0.0.0
- 安全靠防火墙不放行端口已足够
- 切换监听地址实际上跟"加防火墙规则"等价,选简单的

## 后果

### 好处

- xray 配置始终保持干净,没"挂了但还留着"的废条目
- 默认入口端口冲突有兜底,不打扰别人正在跑的生产
- 自启检查闭环,VPS 重启后 xray 不会丢
- "直进直出"判定标准明确,实现不会卡在"算不算"的灰色地带
- 共享 outbound 场景靠现有工具自然兜住,不引入新工具复杂度

### 引入的新约束

- `test/xray_worker/spec.md` 多处改(v4 → v5):
  - §1 不变量"必须有默认入口 18440" → 改"必须有一条 socks5→freedom inbound,端口按让步规则"
  - §3 三分支前置补"看自启"
  - §3 统一收尾流程加"直进直出判定 + 没有就 append"和"内 ping 不通 remove 三件套"
  - §5 端口规则加"默认入口让步算法"
  - §8 加"内 ping 不通时 remove 三件套"唯一例外
  - §三 修订历史加 v5 条目
- `xray/manager.py` 增 `is_enabled()` 方法(T-03 阶段占位,T-07 阶段真实现)
- `task/waiting_03_*.md` 加 `is_enabled()` 占位条目
- 新增 `task/waiting_08_*.md` 内 ping / 外 ping 搬 toolbox + XrayManager wrapper
- `task/waiting_07_*.md` 按 spec v5 改实现轮廓 + 测试用例

### 风险

- **共享 outbound 全不通误判** —— 多 inbound 同时配错账密时会被判"上游死"整组删。用户已接受
- **端口让步降到 1024 仍占满** —— 极罕见。标"任务失败 last_error_code='no_default_port'",等人介入(不退 well-known 段)
