README / deploy / ADR 汇总

vps-server 部署流程图

按客户端快速拉起、首次初始化、不同操作系统守护拉起三段拆开。顶部从左到右选择一种方式，下面按部署步骤从上到下执行。

两进程心智模型

  • mcp_server.py前台收单，由客户端 stdio 拉起，或由 HTTP 服务监听端口。
  • main.py worker-loop后台常驻，消费 task 表并调度 XrayWorker / ProxyDeployWorker。
  • init-db / init-probe-vps首次部署必跑，后续 schema 变化或测试机异常时重跑。

推荐总顺序

1. 准备项目Python 3.10+、uv、git、拉仓库、.env 和加密密钥。
2. 选客户端OpenClaw+GPT 走 HTTP 目标形态；Claude Desktop / Claude Code 走 stdio。
3. 初始化 DB建 ORM 表，看到 init-db 完成 才继续。
4. 初始化测试机没有测试机时先在 .env 添加 PROBE_VPS，再跑自举。
5. 守护 workerLinux systemd、macOS launchd、Windows NSSM 三选一。

OpenClaw + GPT 认证登录部署方式

OpenClaw 走 /sse，Codex / GPT auth 走 /mcp。两边配置互不继承，所以 admin 和 user 两个壳要各注册一次。

HTTP transport 完成后使用 admin 47180 user 47181
  1. 1

    先确认 HTTP MCP 入口已经落地

    README 标注当前 mcp_server.py 还是 stdio 实现。OpenClaw + Codex 的 HTTP 方式需要后续支持 --http、/sse 和 /mcp。

    如果当前代码还没做 HTTP transport，不要直接按这个标签生产部署；先用 Claude stdio，或先完成 HTTP 改造。
  2. 2

    守护 3 个进程

    HTTP 客户端只连端口，不会替你拉子进程。需要常驻 worker-loop、admin MCP、user MCP。

    目标进程
    PYTHONPATH=. uv run python main.py worker-loop
PYTHONPATH=. uv run python mcp_server.py --role admin --http
PYTHONPATH=. uv run python mcp_server.py --role user --http
  3. 3

    OpenClaw 注册 SSE 端点

    写入 ~/.openclaw/openclaw.json，OpenClaw 只认自己的 mcp.servers。

    ~/.openclaw/openclaw.json
    {
  "mcp": {
    "servers": {
      "vps_admin": { "url": "http://127.0.0.1:47180/sse", "transport": "sse" },
      "vps_user":  { "url": "http://127.0.0.1:47181/sse", "transport": "sse" }
    }
  }
}
  4. 4

    Codex / GPT auth 注册 MCP 端点

    写入 ~/.codex/config.toml。不要把 Codex 写成 /sse，也不要把 admin / user 端口写反。

    ~/.codex/config.toml
    [mcp_servers.vps_admin]
url = "http://127.0.0.1:47180/mcp"
startup_timeout_sec = 120

[mcp_servers.vps_user]
url = "http://127.0.0.1:47181/mcp"
startup_timeout_sec = 120
  5. 5

    重启客户端并验证工具可见

    重启 OpenClaw 和 Codex。admin 侧应能看到 register_vps、register_ip、init_db、init_probe_vps；user 侧只暴露查询工具。

    生产部署时 MCP 监听地址必须绑 127.0.0.1，不要把 admin 端点暴露到公网。

Claude Desktop / Claude Code 的 stdio 部署方式

stdio 客户端会按配置自己 fork 子进程，所以 mcp_server.py 不进 systemd / launchd / NSSM；只需要单独守护 main.py worker-loop。

当前代码可用 客户端拉 MCP 只守护 worker
  1. 1

    准备项目路径和虚拟环境

    安装 uv、拉仓库、同步依赖，并确认 .venv/bin/python 或 Windows 的 .venv\\Scripts\\python.exe 存在。

    macOS / Linux
    curl -LsSf https://astral.sh/uv/install.sh | sh
git clone <仓库地址> vps_server
cd vps_server
uv sync
  2. 2

    写 Claude MCP 配置

    Claude Desktop 配置文件通常在 ~/Library/Application Support/Claude/claude_desktop_config.json。Claude Code 使用同样的 stdio 思路。

    stdio config
    {
  "mcpServers": {
    "vps_admin": {
      "command": "/path/to/vps_server/.venv/bin/python",
      "args": ["mcp_server.py", "--role", "admin"],
      "cwd": "/path/to/vps_server",
      "env": { "PYTHONPATH": "/path/to/vps_server" }
    },
    "vps_user": {
      "command": "/path/to/vps_server/.venv/bin/python",
      "args": ["mcp_server.py", "--role", "user"],
      "cwd": "/path/to/vps_server",
      "env": { "PYTHONPATH": "/path/to/vps_server" }
    }
  }
}
  3. 3

    不要守护 stdio MCP server

    Claude 启动时拉起 mcp_server.py，Claude 关闭时回收。把 stdio MCP server 放进守护管理器会抢同一条 stdio 通道。

    只守护 main.py worker-loop。否则 register_vps / register_ip 返回 queued 后，没有后台 worker 消费任务。
  4. 4

    重启 Claude 并验证工具

    重启客户端后，admin 侧应看到登记和初始化工具，user 侧应看到查询工具。首次部署继续执行“初始化 DB”和“初始化测试机”两个标签。

初始化数据库

首次部署必跑。它只负责建表，已有表不会被覆盖；新增 ORM 表后也需要重跑。

幂等 CLI 或 admin MCP
  1. 1

    准备 .env 和加密密钥

    ENCRYPTION_KEY 是生产命脉，同一个 DB 换机器部署必须复用同一把 key。

    .env 初始化
    cp .env.example .env
chmod 600 .env
python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
  2. 2

    先跑测试确认环境

    README 推荐先跑 mock 测试，确认 Python 依赖和导入路径正常。

    测试
    PYTHONPATH=. VPS_SERVER_TESTING=1 uv run pytest -q
  3. 3

    执行 init-db

    看到 init-db 完成: N 张表已就绪 就算成功。

    CLI
    PYTHONPATH=. uv run python main.py init-db
    如果已经配置好 admin MCP，也可以让 agent 调 init_db 工具。
  4. 4

    出错时按错误类型处理

    no such table 基本就是漏跑 init-db；SQLite dev schema 改坏时按 README 的说明手动处理，生产 MySQL 则按迁移策略处理。

初始化测试机

测试机给 IPProbeWorker 校验上游 IP 用，不进业务 VPS 表。没有测试机时先添加 PROBE_VPS_1_* 到 .env。

幂等 默认 inbound 19000 admin only
  1. 1

    添加测试 VPS 凭据

    最多可配 3 台。最小可用配置是 host、port、user、password。

    .env
    PROBE_VPS_1_IP=<测试机公网 IP>
PROBE_VPS_1_PORT=22
PROBE_VPS_1_USER=root
PROBE_VPS_1_PWD=<SSH 密码>
  2. 2

    执行测试机自举

    自举会 SSH 连接测试机，确认或安装 xray，启动服务，并确保 19000 socks5 noauth → freedom inbound 存在。

    CLI
    PYTHONPATH=. uv run python main.py init-probe-vps
# 多台测试机时可选 slot
PYTHONPATH=. uv run python main.py init-probe-vps --slot 2
  3. 3

    确认完成日志

    看到 init-probe-vps 完成: host=... inbound_port=19000 才继续登记 IP。

    如果 agent 查询或登记时返回 probe_vps_not_ready，优先重跑这一页的命令。
  4. 4

    理解它不进 worker 体系

    测试机不是生产代理池资产，不写 vps_record、不派 vps_task，也不走 XrayWorker 的三分支纳管流程。

Linux 用 systemd 拉起常驻 worker

生产推荐 Linux。仓库提供了 deploy/vps-server-worker.service，负责守护 main.py worker-loop。

生产推荐 systemd
  1. 1

    修改 service 里的 4 处占位

    按真实部署路径修改 User、Group、WorkingDirectory、EnvironmentFile。SQLite 场景还要确认 ReadWritePaths 可写。

  2. 2

    安装并启动 worker 服务

    systemd
    sudo cp deploy/vps-server-worker.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now vps-server-worker
  3. 3

    验证状态和日志

    检查
    sudo systemctl status vps-server-worker
sudo journalctl -u vps-server-worker -f
    应看到 main.worker-loop 启动，没任务时会静默轮询。
  4. 4

    HTTP 客户端额外守护 MCP admin / user

    只用 Claude stdio 时不需要这一步。OpenClaw + Codex HTTP 模式下，HTTP 改造完成后还要按同样思路新增 admin 47180 和 user 47181 两个 systemd unit。

    当前仓库只提供 worker 的 systemd 文件；MCP HTTP unit 需要在 HTTP transport 落地后补齐。

macOS 用 launchd 拉起常驻 worker

适合开发机或个人测试。仓库提供 deploy/com.vps-server.worker.plist。

launchd 不建议生产
  1. 1

    修改 plist 路径占位

    把 /Users/YOU/path/to/vps_server 换成实际项目路径，同时修改 python 路径、PYTHONPATH 和日志路径。

  2. 2

    安装到用户级 LaunchAgents

    launchd
    cp deploy/com.vps-server.worker.plist ~/Library/LaunchAgents/
launchctl load -w ~/Library/LaunchAgents/com.vps-server.worker.plist
  3. 3

    验证运行状态

    检查
    launchctl list | grep vps-server
tail -f /tmp/vps-server-worker.log
  4. 4

    停止和取消注册

    卸载
    launchctl unload -w ~/Library/LaunchAgents/com.vps-server.worker.plist
    macOS 盖电脑会挂起进程，开发可用，生产优先放 Linux。

Windows 用 NSSM 拉起常驻 worker

NSSM 把 Python 命令包成 Windows 服务，仓库的详细说明在 deploy/install-windows-nssm.md。

NSSM SERVICE_AUTO_START
  1. 1

    安装 NSSM

    PowerShell
    choco install nssm
# 或下载 https://nssm.cc/download 并把 win64 目录加入 PATH
  2. 2

    注册 worker 服务

    PowerShell
    $ProjectRoot = "C:\Path\To\vps_server"
$PythonExe   = "$ProjectRoot\.venv\Scripts\python.exe"

nssm install vps-server-worker $PythonExe "main.py worker-loop"
nssm set vps-server-worker AppDirectory        $ProjectRoot
nssm set vps-server-worker AppEnvironmentExtra "PYTHONPATH=$ProjectRoot"
nssm set vps-server-worker AppStdout           "$ProjectRoot\logs\worker.log"
nssm set vps-server-worker AppStderr           "$ProjectRoot\logs\worker.err"
nssm set vps-server-worker Start               SERVICE_AUTO_START
nssm set vps-server-worker AppExit Default     Restart
nssm set vps-server-worker AppRestartDelay     5000
  3. 3

    启动并查看日志

    PowerShell
    nssm start vps-server-worker
nssm status vps-server-worker
Get-Content "$ProjectRoot\logs\worker.log" -Tail 50 -Wait
  4. 4

    HTTP 客户端额外注册 MCP 服务

    只用 Claude stdio 时不需要。OpenClaw + Codex HTTP 模式下，HTTP 改造完成后按 deploy/install-windows-nssm.md 第 2 节注册 vps-server-mcp-admin 和 vps-server-mcp-user。

    NSSM 停服务时会先模拟 Ctrl+C，和 main.py 的 SIGINT handler 对齐。
