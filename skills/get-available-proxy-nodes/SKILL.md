# 查询可用代理节点（user 侧）

用 user 侧 MCP 的唯一工具 `get_available_proxy_nodes` 查询可用代理节点，整理成用户可直接填入客户端的格式交付。

## 边界

- user 侧 MCP **只有这一个工具**，纯只读。
- 登记 VPS、登记 IP、初始化 Xray、改到期日等管理动作全部在 admin 侧 MCP。
  用户提出这类需求时，不要尝试用本工具凑，直接回答：请联系管理员处理。
- 节点数据的唯一来源是本工具的返回结果。数据库之外的节点不存在，不要编造。

## 工具参考

### `get_available_proxy_nodes`

**用途：** 列出当前可用的代理节点，可按地区过滤。

**参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `country_code` | string | 否 | 大写 ISO 两字母地区代码，例 `SG` / `US` / `JP`。不传或传空串 = 返回全部可用节点 |

**调用方式：**

- 用户指定了地区 → 传 `country_code`（口语先转码：新加坡→SG、美国→US、日本→JP、香港→HK）
- 用户没指定地区 → 不传参数

**返回形状：** JSON 数组，每个元素是一个节点：

```json
[
  {
    "proxy_id": 1,
    "vps_id": 1,
    "ip_id": 1,
    "protocol": "socks5",
    "host": "203.0.113.10",
    "port": 18441,
    "username": "xxx",
    "password": "yyy",
    "egress_ip": "198.51.100.10",
    "country_code": "SG",
    "country_name": "Singapore",
    "city": "Singapore"
  }
]
```

无匹配时返回 `[]`。

**「可用」的定义（后端已过滤，不需要再判断）：**
返回的每个节点都满足：代理状态正常使用中、所在 VPS 和上游 IP 都未过期未停用。
拿到的就是能用的，不要自己再筛一遍。

## 交付格式

不要直接甩 JSON。整理成小火箭（Shadowrocket）可手动填写的格式：

```
类型：socks5
IP：203.0.113.10
端口：18441
账号：xxx
密码：yyy
出口IP：198.51.100.10
地区：新加坡 Singapore
```

字段对应关系：类型=`protocol`、IP=`host`、端口=`port`、账号=`username`、
密码=`password`、出口IP=`egress_ip`、地区=`country_name`（+`city`）。

## 示例

### 例 1：用户指定地区

> 用户：给我一个新加坡节点

1. 调 `get_available_proxy_nodes`，传 `country_code="SG"`
2. 返回 1 个节点 → 直接按交付格式整理给用户
3. 返回多个节点 → 用户只要 1 个，取列表第一个给用户

### 例 2：用户没指定地区

> 用户：现在有哪些节点能用？

1. 调 `get_available_proxy_nodes`，不传参数
2. 用户问的是「有哪些」→ 全部列出，按地区分组更易读

### 例 3：查不到节点

> 用户：有没有韩国节点？

1. 调 `get_available_proxy_nodes`，传 `country_code="KR"`
2. 返回 `[]` → 回答：「当前没有韩国地区的节点，需要找管理员添加。」
3. 不要拿别的地区节点冒充，也不要承诺「稍后会有」。

## 反例（禁止事项）

- ❌ 未调用工具就承诺「有节点」「马上给你」——先查再答
- ❌ 返回空数组时编造节点，或把数据库外的节点告诉用户
- ❌ 直接把原始 JSON 甩给用户——必须整理成交付格式
- ❌ 用本工具登记 VPS、新增出口 IP、初始化 Xray、检查端口——user 侧没有这些能力，引导用户找管理员
- ❌ 用户要 1 个节点却把全部节点连账密一起倒出来——按需交付
